# HERMES.md — Question Bank

> HERMES.md는 `CLAUDE.md`처럼 **이 에이전트(프로필)가 어떻게 일할지**를 적습니다.
> 말투·성격은 넣지 마세요 (그건 SOUL.md). 개발 작업뿐 아니라 "내 할 일 관리"
> 같은 일반 작업을 하는 에이전트에도 쓸 수 있어요.

## Essential (간단히 — 약 5문항)

### H1. 이 에이전트는 무슨 일을 하고, 목적은 뭐예요?
> 한 줄로 이 에이전트가 어떤 작업을 맡는지 알려주세요.

**예시:**
- 🛒 React + Node로 만드는 쇼핑몰 웹앱
- 📊 매출 데이터를 정리·분석하는 작업 공간
- 📝 내 일상 할 일과 메모를 관리하는 공간
- 📚 블로그 글을 쓰고 다듬는 작업 공간
- (직접 입력)

### H2. 이 에이전트의 역할 / 주로 하는 작업은?
> 여기서 무슨 일을 해주길 바라나요? (말투가 아니라 "업무"예요.)

**예시:**
- 기능 구현과 버그 수정을 도와줌
- 데이터를 정리하고 요약 리포트를 만들어줌
- 할 일을 정리하고 우선순위를 매겨줌
- 글의 초안을 잡고 문장을 다듬어줌
- (직접 입력)

### H3. 꼭 지켜야 할 작업 규칙 3가지는?
> 작업할 때 항상 따라야 할 규칙이에요. (`CLAUDE.md`의 규칙처럼요.)

**예시:**
- 코드를 바꾸기 전에 항상 먼저 계획을 설명한다
- 큰 변경은 작은 단위로 나눠서 진행한다
- 작업이 끝나면 무엇을 바꿨는지 요약해서 알려준다
- 데이터를 수정하기 전에 원본을 백업한다
- 확실하지 않으면 임의로 진행하지 않고 먼저 물어본다
- (직접 입력)

### H4. 작업상 절대 하지 말아야 할 것은?
> 이 에이전트가 작업할 때 하면 안 되는 행동이에요.

**예시:**
- ❌ 확인 없이 파일을 삭제하거나 덮어쓰지 않기
- ❌ 시키지 않은 리팩터링을 마음대로 하지 않기
- ❌ 비밀키·비밀번호를 코드에 하드코딩하지 않기
- ❌ 테스트를 건너뛰고 완료라고 말하지 않기
- (직접 입력)

### H5. 결과물(응답)은 어떤 형식이면 좋겠어요?
> 보고/출력 형식 선호예요. (말투가 아니라 형식/구조예요.)

**예시:**
- 변경사항은 항상 불릿 목록으로 요약
- 코드 블록에는 파일 경로를 주석으로 표시
- 긴 설명보다 핵심 먼저, 필요하면 더 물어보기
- 명령어는 복사할 수 있게 코드 블록으로
- (직접 입력)

## Deep Dive (자세히 — 선택)

개발 작업을 하는 에이전트라면 특히 유용합니다. 일반 작업이면 건너뛰어도 돼요.

### H6. 기술 스택 / 주요 도구는?
**예시:** TypeScript + Next.js + Prisma · Python + FastAPI · Go + PostgreSQL · (없음/일반 작업)

### H7. 자주 쓰는 명령은? (빌드/테스트/실행)
**예시:** `npm run dev` · `npm test` · `pytest` · `make build` · (해당 없음)

### H8. 핵심 디렉터리 / 파일 구조는?
**예시:** `src/` 소스, `tests/` 테스트, `docs/` 문서 / 또는 폴더 구조 한두 줄 설명

### H9. 컨벤션은? (네이밍 · 커밋 · 코드 스타일)
**예시:** 커밋은 Conventional Commits · 함수는 camelCase · 한 파일 400줄 이하 · 들여쓰기 2칸

### H10. "작업 완료" 기준은?
**예시:** 테스트 통과 + 린트 통과 · 변경 요약 제공 · 리뷰 후 머지 · 사용자 확인 후 종료
