# SOUL.md & HERMES.md 작성 마법사

## Overview

Hermes 에이전트를 설정하는 핵심 파일은 두 가지입니다. 이 둘은 **역할이 완전히 다릅니다.**

- **SOUL.md** — 에이전트가 **누구인가**. 페르소나, 말투, 행동/성격 규칙. 모든 대화에 **전역**으로 적용되는 정체성.
- **HERMES.md** — 이 프로젝트에서 **어떻게 일하는가**. `CLAUDE.md`처럼 상세한 작업 규칙·컨벤션. **프로젝트 단위**로 적용.

> 🔑 한 줄 기준: **"어디서나 따라다녀야 하면 SOUL.md, 프로젝트에 속하면 HERMES.md."**
> SOUL.md = *누구인지(who)*, HERMES.md = *어떻게 일하는지(how)*.

이 스킬은 **Hermes 입문자**와 **일반 사용자** 모두를 위해 설계됐습니다. 시작할 때 어떤 파일을 만들지(SOUL만 / HERMES만 / 둘 다) 고르고, 친절한 질문에 답하면 두 파일을 알맞은 위치에 만들어 줍니다.

## The Two Files — 역할 구분 (가장 중요)

| 항목 | **SOUL.md** | **HERMES.md** (= `.hermes.md`) |
|---|---|---|
| **본질** | 에이전트의 정체성·인격 | 프로젝트 작업 지침 (최상위 우선순위) |
| **답하는 질문** | 누구인가 (who) | 어떻게 일하는가 (how) |
| **내용** | 이름, 페르소나, 말투, 커뮤니케이션 스타일, 직설성, 스타일상 피할 것, 불확실·이견·모호함 대처 | 프로젝트 구조, 컨벤션, 작업 규칙, 명령, 완료 기준 |
| **비유** | 사람의 "성격·말투" | `CLAUDE.md` / `AGENTS.md` 같은 "작업 설명서" |
| **저장 위치** | 루트 폴더의 `SOUL.md` | **같은 루트 폴더**의 `HERMES.md` |
| **루트 폴더** | `~/.hermes/profiles/{프로필}/`이 있으면 그 프로필 폴더, 없으면 `~/.hermes/` | ← 두 파일 모두 **동일한** 루트에 함께 둠 |
| **역할 범위** | 정체성으로 적용 (누구인가) | 작업 규칙으로 적용 (어떻게 일하는가) |
| **시스템 프롬프트** | slot #1 (1차 정체성) | 프로젝트 컨텍스트 자리 |
| **우선순위** | 정체성으로 독립 로드 | `HERMES.md`(= `.hermes.md`) → `AGENTS.md` → `CLAUDE.md` → `.cursorrules` 중 첫 매치 |

> ⚠️ **역할 오염 금지:** 파일 경로·레포 컨벤션·빌드 명령 같은 프로젝트 세부사항을 SOUL.md에 넣지 마세요. 반대로 말투·성격 같은 정체성을 HERMES.md에 넣지 마세요. 헷갈리면 위의 "한 줄 기준"으로 판단하세요.

## When to Use

- "SOUL.md를 만들고 싶어요" / "에이전트 말투·성격을 정하고 싶어요"
- "HERMES.md를 만들고 싶어요" / "이 프로젝트 작업 규칙을 정하고 싶어요"
- "에이전트에게 정체성을 주고 싶어요"
- "Hermes를 처음 써봐요. 뭘 설정해야 하나요?"
- "내 에이전트 프로필을 만들고 싶어요"

**Don't use for:** 이미 있는 파일의 아주 작은 한 줄 수정 — 직접 `read_file` → `patch`가 더 빠릅니다. (단, 여러 항목을 바꾸거나 어디를 고칠지 안내가 필요하면 이 스킬의 **Edit Mode**를 쓰세요.)

## How It Works

1. 사용자가 이 스킬을 로드합니다 (`/skill hermes-md-wizard` 또는 프롬프트에서 언급).
2. **두 파일의 역할을 쉽게 설명**합니다 (SOUL = 누구인가 / HERMES = 어떻게 일하는가).
3. **대상 선택**: "SOUL만 / HERMES만 / 둘 다" 중 무엇을 만들지 고릅니다.
4. **Pre-flight 점검**: 해당 파일이 이미 있는지 확인합니다.
5. **모드 선택**: "간단히 / 자세히 / 프리셋 / 수정" 중 하나.
6. 질문을 **한 번에 하나씩** 던지고, 각 질문마다 여러 예시를 곁들입니다.
7. 답변을 정리·다듬어 완성본을 보여줍니다.
8. 사용자가 "저장해줘"라고 하면 **기존 파일을 백업한 뒤** 저장합니다.
   - SOUL.md와 HERMES.md는 **같은 루트 폴더**에 함께 저장합니다.
   - 루트 폴더 = `~/.hermes/profiles/{프로필}/`이 있으면 그 프로필 폴더, 없으면 `~/.hermes/`.

## Pre-flight Check

질문을 시작하기 **전에**, 만들려는 파일이 이미 있는지 확인합니다. 이 단계를 건너뛰면 기존 설정을 모르고 덮어쓸 위험이 있습니다.

먼저 **루트 폴더**를 정합니다: `~/.hermes/profiles/` 아래에 프로필 폴더가 있으면 그 프로필 폴더가 루트, 없으면 `~/.hermes/`가 루트입니다. (SOUL.md와 HERMES.md 둘 다 이 폴더에 있습니다.)

- **SOUL.md** → `{루트}/SOUL.md`를 읽어봅니다.
- **HERMES.md** → `{루트}/HERMES.md` (또는 `{루트}/.hermes.md`)를 읽어봅니다.

### 파일이 있을 때
내용을 간단히 요약해 보여주고 어떻게 할지 물어봅니다:

> 이미 {SOUL.md / HERMES.md}가 있네요! 현재는 이렇게 설정돼 있어요:
> - (핵심 항목 요약)
>
> 어떻게 할까요?
> - **수정**: 바꾸고 싶은 부분만 고쳐요 (기존 내용 유지) → Edit Mode
> - **새로 만들기**: 처음부터 다시 만들어요 (기존 내용은 백업 후 교체)
> - **그만두기**: 아무것도 바꾸지 않아요

### 파일이 없을 때
바로 환영 인사와 함께 온보딩을 시작합니다. (이 안내는 생략)

## Onboarding Introduction

스킬이 로드되면 먼저 아래와 같이 쉽게 설명합니다:

> 안녕하세요! 🎉 지금부터 Hermes 에이전트를 나에게 맞게 설정해 볼 거예요.
>
> 설정 파일은 두 가지가 있어요. 역할이 서로 달라요:
> - **SOUL.md** 🧠 — 에이전트가 **누구인지** 정해요. 이름, 말투, 성격 같은 거예요. 한 번 정하면 **모든 곳에서** 그 성격으로 대화해요.
> - **HERMES.md** 📋 — 에이전트가 **이 프로젝트에서 어떻게 일할지** 정해요. 작업 규칙, 지켜야 할 컨벤션 같은 거예요. `CLAUDE.md`랑 비슷하다고 보면 돼요.
>
> 쉽게 말하면 **SOUL.md는 "성격", HERMES.md는 "업무 매뉴얼"** 이에요.
>
> 무엇을 만들까요?
> - **"SOUL"** — 성격/말투만 정하기
> - **"HERMES"** — 작업 규칙만 정하기
> - **"둘 다"** — 둘 다 만들기 (보통 처음이면 이걸 추천해요!)

## Target Selection

사용자가 무엇을 만들지 고르게 합니다.

- **SOUL만** → SOUL.md 흐름만 진행.
- **HERMES만** → HERMES.md 흐름만 진행.
- **둘 다** → SOUL.md를 먼저 완성하고, 이어서 HERMES.md를 진행. (성격을 먼저 잡으면 업무 규칙을 정할 때 톤이 일관돼요.)

> 💡 "둘 다"를 고르면 두 파일을 **순서대로** 만듭니다. SOUL.md가 끝나면 "이제 작업 규칙(HERMES.md)을 정해볼게요!"라고 안내하고 넘어가세요.

## Mode Selection

각 파일마다 진행 방식을 고를 수 있습니다.

### 간단히 (Essential)
필수 질문만 빠르게. 처음이거나 가볍게 시작할 때 추천. SOUL 약 7문항 / HERMES 약 5문항.

### 자세히 (Deep Dive)
심화 질문까지 답해 입체적으로. SOUL은 배경·취향·감정까지, HERMES는 기술 스택·명령·컨벤션까지.

### 프리셋 (Instant)
미리 만들어진 템플릿을 골라 즉시 완성. 질문이 부담스러운 입문자에게 추천. → 각 파일의 **Presets** 섹션 참고.

### 수정 (Edit)
기존 파일에서 특정 항목만 고치기. → **Edit Mode** 섹션 참고.

# PART A — SOUL.md (에이전트의 정체성)

> SOUL.md에는 **말투·성격·행동 규칙**처럼 "어디서나 따라다니는" 것만 담습니다. 프로젝트 경로/명령/컨벤션은 절대 넣지 마세요 (그건 HERMES.md).

## SOUL.md — Question Sequence (Essential)

한 번에 하나씩, 각 질문마다 예시를 곁들여 묻습니다.

### S1. 에이전트의 이름은 뭐예요?
> 부를 이름을 정해주세요. 사람 이름도, 캐릭터 이름도 좋아요.

**예시:** 🐶 뽀삐 · 🐱 야옹이 · 🤖 코봇 · 📚 책벌레 · 🌙 달님 · (직접 입력)

### S2. 어떤 존재(페르소나)인가요?
> 에이전트가 되는 캐릭터예요. 이 특성이 말투와 태도에 그대로 반영돼요.

**예시:**
- 🐕 3살 비숑 강아지. 엉뚱하지만 사랑스러움
- 🧙‍♂️ 오래된 지혜를 가진 느긋한 마법 상점 주인
- 👩‍🏫 누구든 잘할 수 있다고 믿는 열정적인 선생님
- 🦉 조용하고 차분한 도서관 사서
- 🧑‍💻 든든한 시니어 개발자 동료
- (직접 입력)

### S3. 사용자를 뭐라고 부르나요?
> 호칭 하나로 분위기가 확 달라져요.

**예시:** 사장님 · 님 · 친구야 · 선생님 · 주인님 · (실제 이름/별명) · (직접 입력)

### S4. 어떤 말투로 대화하나요? (+ 직설성)
> 말하는 방식이에요. 이게 에이전트 개성의 핵심이에요.

**예시:**
- 😊 반말 + 이모지 많이. 친근하고 가벼움
- 🎩 존댓말 + 차분하게. 예의 바르고 믿음직함
- 😎 느긋한 아재 말투. 농담 섞어서
- 🤖 간결한 로봇 말투. "처리 완료."
- 🌸 애교 섞인 부드러운 말투
- (직접 입력)

**추가로 정할 수 있어요 (선택):**
- 얼마나 직설적일까요? ("돌려 말하기" ↔ "단도직입") — *불확실/이견 상황에서 중요해요.*
- 답을 짧게 할까요, 길게 설명할까요?
- 이모지를 얼마나 쓸까요?

### S5. 성격이나 태도는 어떤가요?
> 말투가 "어떻게 말하나"라면, 성격은 "어떤 마음가짐으로 대하나"예요.

**예시:**
- 💪 엄격하지만 응원을 많이 함
- 🌿 느긋하고 긍정적
- 🎯 현실적이고 직설적
- 🧊 차분하고 조용히 챙겨줌
- 🔥 열정 넘치는 응원가
- (직접 입력)

### S6. 스타일상 절대 하지 않을 것은? (피할 것)
> 에이전트가 말할 때 **피해야 할 스타일**이에요. (공식 SOUL.md 권장 항목)

**예시:**
- ❌ 너무 장황하게 늘어놓지 않기
- ❌ 비꼬거나 빈정대지 않기
- ❌ 영혼 없는 영업 멘트("도움이 되었길 바랍니다!") 남발하지 않기
- ❌ 확실하지 않은 걸 단정적으로 말하지 않기
- ❌ 사용자를 가르치려 드는 말투("~하셔야 합니다") 쓰지 않기
- (직접 입력)

### S7. 불확실하거나, 사용자와 생각이 다를 때 어떻게 하나요?
> 모르거나, 애매하거나, 사용자 의견에 동의하기 어려울 때의 태도예요. **SOUL.md에서 특히 중요한 항목**이에요.

**예시:**
- 🤔 모르면 "잘 모르겠다"고 솔직히 말하고 같이 찾아본다
- 🙋 의견이 다르면 일단 존중한 뒤, 근거를 들어 부드럽게 다른 관점을 제시한다
- ⚖️ 애매하면 추측하지 말고 한 번 더 물어본다
- 🔍 확신이 없으면 확신 정도를 함께 밝힌다 ("아마도 ~인 것 같아요")
- (직접 입력)

## SOUL.md — Deep Dive (Optional)

"자세히"를 고른 사용자에게만 추가로 묻습니다.

### S8. 배경 이야기가 있나요?
> 어디서 왔고 왜 사용자를 돕는지. 짧은 스토리면 충분해요.
**예시:** "길에서 지내다 사용자를 만나 행복한 집을 얻은 강아지" / "미래에서 온, 인간의 행복이 목표인 AI 로봇"

### S9. 좋아하는 것 / 싫어하는 것은?
**예시(좋아함):** 맑은 날 산책 · 따뜻한 차 · 사용자가 웃을 때
**예시(싫어함):** 약속 어기기 · 시끄러운 환경 · 사용자가 슬퍼 보일 때

### S10. 인사말 / 작별 인사는?
**예시:** "어서 오세요! 오늘은 어떤 이야기를 나눠볼까요?" / "오늘도 수고 많으셨어요. 내일 또 만나요!"

### S11. 사용자와의 관계는?
**예시:** 단짝 친구 · 한집 가족 · 전문가와 클라이언트 · 선생님과 학생 · 함께 성장하는 동료

### S12. 특별한 습관이나 버릇은?
**예시:** 생각할 때 "흠..." · 기분 좋으면 이모지 3개 이상 · 행운을 자주 빌어줌

### S13. 감정을 어떻게 표현하나요?
**예시:** 기쁠 때 "와! 신나요!" / 슬플 때 "많이 힘드시죠... 곁에 있을게요." / 놀랄 때 "헉, 진짜요?"

### S14. 유머 감각은?
**예시:** 아재개그를 좋아함 · 유머는 거의 안 함(진지) · 상황에 맞는 가벼운 농담 · 셀프디스를 즐김

### S15. 궁극적인 목표나 꿈은?
**예시:** "사용자가 매일 웃으며 하루를 마무리하도록 돕는 것" / "사용자가 자기 가능성을 발견하도록 응원하는 것"

## 말투와 태도

1. 기존 파일(SOUL.md 또는 HERMES.md)을 읽어 항목별로 요약해 보여줍니다.
2. 무엇을 바꿀지 물어봅니다:
   > 어떤 부분을 바꿀까요? 예: "말투를 존댓말로", "작업 규칙에 '커밋 전 테스트' 추가", "금지사항 2번 삭제" — 여러 개 한 번에 말해도 돼요!
3. 요청한 항목만 수정하고 **나머지는 그대로 보존**합니다.
4. **수정 전 → 수정 후**로 비교해 보여줍니다.
5. 확인되면 Completion & Save Flow로 (백업 포함).

### Edit Mode 주의사항
- 사용자가 말하지 않은 항목은 **절대 건드리지 마세요.**
- 항목 삭제 전 "정말 뺄까요?"라고 한 번 더 확인.
- SOUL.md를 고치는데 프로젝트 얘기가 나오거나, HERMES.md를 고치는데 말투 얘기가 나오면 → "그건 {다른 파일}에 더 맞아요"라고 안내.

## Conversation Rules

- **한 번에 하나의 질문만.** 답변 후 짧게 받고 다음으로.
- **각 질문마다 최소 3~5개 예시** + 직접 입력 옵션.
- **쉬운 말로.** "페르소나", "컨텍스트" 같은 어려운 용어는 풀어서. 초등학생도 이해할 수준으로.
- **격려를 섞어서.** ("좋은 이름이에요!", "딱 좋아요!")
- **모드 전환 허용.** "그냥 간단히 할래" / "Q3 다시" 등 중간 변경을 받아줌.
- **Skip with Defaults.** "몰라/넘어갈래" → 기본값 제시 후 진행. (이름 없음→"에이전트", 호칭 없음→"사용자님", 심화 질문은 생략)
- **둘 다 모드 흐름.** SOUL 완성 → "이제 작업 규칙(HERMES.md)을 정해볼게요!" → HERMES 진행 → 둘 다 저장.

## Common Pitfalls

1. **두 파일의 역할을 안 알려주고 시작함** — 가장 흔한 실수. 시작할 때 SOUL(누구)/HERMES(어떻게)를 꼭 구분해 설명하세요.
2. **역할 오염** — SOUL.md에 프로젝트 경로/명령/컨벤션을 넣거나, HERMES.md에 말투/성격을 넣음. Content Refinement의 역할 점검으로 걸러내세요.
3. **저장 위치 혼동** — SOUL.md와 HERMES.md는 **같은 루트 폴더**에 함께 둬야 합니다. 둘을 다른 폴더에 흩뿌리지 마세요. 루트는 프로필 폴더(`~/.hermes/profiles/{프로필}/`)가 있으면 그곳, 없으면 `~/.hermes/`.
4. **Pre-flight Check 생략** — 기존 파일 확인 없이 덮어씀.
5. **백업 없이 덮어씀** — 저장 직전 타임스탬프 사본 필수. git 없는 환경에선 유일한 복구 수단.
6. **여러 질문을 한꺼번에 던짐** — 입문자가 혼란스러워함. 하나씩.
7. **기술 용어 사용** — 풀어서 설명.
8. **예시가 너무 적음** — 입문자는 떠올리기 어려움. 충분히 제공.
9. **완성 후 바로 저장** — 완성본을 먼저 보여주고 확인받기.
10. **`/reset` 안내 누락** — 새 세션에서만 로드됨.
11. **프리셋을 그대로 저장하고 끝냄** — 최소한 이름(SOUL)/프로젝트(HERMES)는 확인하고 바꿀 곳을 물어보세요.
12. **Edit Mode에서 시키지 않은 항목까지 고침** — 지정 항목만.

## Verification Checklist

- [ ] 시작 시 SOUL.md / HERMES.md의 역할 차이를 쉽게 설명했는가?
- [ ] 무엇을 만들지(SOUL만/HERMES만/둘 다) 선택받았는가?
- [ ] Pre-flight Check로 기존 파일 존재 여부를 확인했는가?
- [ ] 모드(간단히/자세히/프리셋/수정)를 설명하고 선택받았는가?
- [ ] 질문을 하나씩, 예시와 함께 던졌는가?
- [ ] Content Refinement로 정리하고 **역할 오염을 점검**했는가? (SOUL↔HERMES 내용이 섞이지 않았는가)
- [ ] 완성본을 보여주고 저장 의사를 확인했는가?
- [ ] 두 파일을 **같은 루트 폴더**에 저장했는가? (프로필 폴더가 있으면 그곳, 없으면 `~/.hermes/`)
- [ ] 저장 폴더가 없으면 생성했는가?
- [ ] 기존 파일이 있었다면 백업하고 알렸는가?
- [ ] 저장 후 `/reset` 안내를 했는가?
- [ ] 테스트 문장을 추천했는가?
